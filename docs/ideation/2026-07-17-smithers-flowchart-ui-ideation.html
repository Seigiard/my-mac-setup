Flow-chart UI для smithers-воркфлоу (se-pipeline)

date: 2026-07-17 topic: smithers-flowchart-ui mode: repo-grounded

Focus: открыть интерфейс и (а) просматривать/настраивать флоу — какие шаги какими агентами выполняются, какие развилки решений есть; (б) там же видеть запущенные флоу как flow-chart с подсветкой текущего этапа и ценами за выполненные шаги.

Grounding Context

Codebase Context. se-pipeline — durable-конвейер (verify-doc → work → secret-scan → verify-code → rescan) на smithers-orchestrator 0.28; воркфлоу — JSX/TSX-деревья, рендерящиеся динамически в рантайме; состояние в sqlite. Все 5 стадий построены одним шаблоном stageBlock() (попытка → гейт green/failed/degraded → Approval-развилка → доп. попытка → stop-only Approval). У smithers уже есть: smithers graph (статический скелет первого рендера — без развилок и Approval-рёбер), smithers monitor/gui (живой web-UI, Gateway на :7331), TUI-режимы, observability (локальный Grafana-стек). Gateway отдаёт готовые React-хуки (useGatewayRunTree с инкрементальной сверкой дерева, useGatewayRunEvents с seq-номерами, useGatewayApprovals + submitApproval) — настоящая подписка, не поллинг. Стоимость: события TokenUsageReported ложатся в _smithers_events по nodeId во время прогона (верифицировано: движок «feeds it live events» для бюджет-трекера); summary-строка хранит stageUsage (разбивка по нодам) и planHash — обе колонки нигде не отображаются. Редактируемость: рантайм-инпуты (validateCmd, таймауты, бюджет work) против запиненных в TSX констант (модели, review-бюджет, waiveOnApprove); правка исходников ломает resume in-flight прогонов (RESUME_METADATA_MISMATCH).

External context. Жёсткий водораздел в prior art: code-first движки (Temporal, Airflow, Dagster, Prefect) рендерят граф только как read-only проекцию кода; двунаправленное редактирование есть у Windmill и LangGraph Studio — оба ценой ограничения авторинга сериализуемым пошаговым форматом, чем runtime-JSX smithers не является. Паттерны: waterfall-бары Inngest (обобщаются на стоимость), трейс-вью LangSmith с per-step cost, OTel GenAI semconv. «Атрибуция — какой шаг/агент сжёг деньги — то, что observability-стеки делают хуже всего».

Past learnings: docs/solutions/ в репо отсутствует. Web-research: свежий диспатч (кэша не было).

Topic Axes

  • static-graph — статический граф воркфлоу: структура, развилки, агенты по шагам
  • live-run-control — live-наблюдение прогона и управление (подсветка стадии, approvals)
  • flow-editing — редактирование/конфигурирование флоу
  • per-step-cost — стоимость по шагам: оверлеи, агрегаты, live

Ranked Ideas

  • 2. Замороженный граф-артефакт прогона со стоимостью по шагам
  • 3. Пульт поверх monitor: push + andon + sparkline

1. Полный граф развилок из stageBlock-шаблона

static-graphConfidence 85%Complexity: Medium

Description: Все пять стадий — вызовы одного шаблона stageBlock() (ровно 5 call-sites: se-pipeline.tsx:478/528/575/606/639) с параметрами-литералами. Экстрактор (~100–200 строк) один раз материализует полный граф: попытка → 3-путевой вердикт гейта (green/failed/degraded) → Approval (approve/deny) → доп. попытка → stop-only Approval — все рёбра, которые smithers graph структурно не может показать (он видит только скелет первого рендера). На нодах — «call sheet»: модель + фолбэк, таймаут, бюджет-предохранитель, waiveOnApprove. Выхлоп — Mermaid/SVG + JSON; CI-проверка, что закоммиченный граф не разъехался с кодом (и с runbook-таблицей семантик approve). Опциональный оверлей: эмпирические частоты рёбер из истории прогонов («approve на work сработал в 4/9 прогонов», неисхоженные ветки — пунктиром).

Где смотреть (решение оператора, поверхности не исключают друг друга): (a) команда se graph — генерит self-contained SVG/HTML и открывает в браузере (паттерн reportDir, ноль инфраструктуры); (b) закоммиченный docs/se-pipeline-graph.html рядом с раннбуком + CI-гейт против дрейфа с кодом; (c) страница поверх monitor — только если жить в нём. Рекомендация: a+b от одного генератора. Приём из custom-harness: layout-хинты (позиции нод) — отдельные персистентные данные рядом с исходником, не автолэйаут на каждый рендер.

Basis
direct: «smithers graph … Does NOT render conditional children (only first-render skeleton) … Does NOT show approval forks» (evidence-graph-structure); 5 call-sites stageBlock верифицированы построчно; параметры — литералы (name, gateFn, waiveOnApprove). Модели/бюджеты: se-pipeline.tsx:104–110, review maxBudgetUsd: 15.
Rationale
Это прямой ответ на обе половины вопроса (а): «какие шаги какими агентами» и «какие развилки есть». Сегодня ответ существует только в голове + TSX. Дыра не в рендеринге, а в том, что единственный штатный инструмент структурно не исследует условное дерево — а авторы конвейера, сами того не заметив, уже ограничили себя сериализуемым шаблоном, так что граф извлекается статически, без исполнения воркфлоу. Один раз построенный, этот граф — подложка для всех остальных идей (live-подсветка, cost-бейджи, артефакт прогона).
Downsides
Экстрактор привязан к конвенции stageBlock — новый воркфлоу «не по шаблону» потребует расширения; эмпирический оверлей требует майнинга _smithers_events (не проверен на масштабе).

2. Замороженный граф-артефакт прогона со стоимостью по шагам

static-graphConfidence 80%Complexity: Medium

Description: UI не обязан быть приложением. Summary-таск (он уже агрегирует usage и пишет отчёты) дополнительно рендерит один self-contained HTML/SVG: граф из идеи №1 с подсвеченным фактическим путём — вердикты гейтов, waives, и $-бейджи по нодам из stageUsage с анфолдом :child:-legs (claude vs opencode ревью — по отдельности, а не одним свёрнутым числом). Токены — первичная метрика, USD — мелко (раннбук: USD — оценка, не биллинг). Бонус-слой: бар «расход против приора» — медианы стадий по прошлым прогонам, аномалия видна на глаз. Файл ложится в reportDir рядом с work.envelope.json, se show его перечисляет; открывается через годы без Gateway и живой версии smithers. Решение оператора: live-тикание НЕ нужно — запись в граф законченного воркфлоу достаточна. Шаг 0 (не дожидаясь графа): распарсить stageUsage в se show — одна строка, разбивка по стадиям появляется в CLI сегодня. Готового артефакта пока не существует; данные для прототипа лежат готовые (summary + stageUsage эталонных прогонов в smithers.db).

Basis
direct: summary персистит stageUsage: JSON.stringify(usage.stages) и planHash; «Report dir and files enumerated from filesystem» в se show; gap-таблица evidence-cost-data («stageUsage … never parsed/displayed»; «child-run cost folding opaque»). external: compound-бары Inngest; custom-harness: SQLite смизерса как единая read-model для UI.
Rationale
Прямой ответ на «цены за выполненные шаги» + пост-мортемы («что реально сделал тот прогон за $8») переживают любой живой UI. Атрибуция — какой лег сжёг бюджет — уже посчитана и лежит мёртвым грузом в колонке. Естественная вторая половина идеи №1: тот же граф, но с историей конкретного прогона; готовая единица сравнения для F3.
Downsides
Не live (снимок на момент завершения — принято оператором сознательно); шаблон надо поддерживать при изменении структуры конвейера (смягчается генерацией из графа №1); приоры требуют нескольких прогонов истории.
Confidence
80%
Complexity
Medium

3. Пульт поверх monitor: push + andon + sparkline

live-run-controlConfidence 65%Complexity: Medium

Description: Ключевая находка проверки: инбокс approvals с consequence-текстом уже встроен в smithers monitor (ApprovalsInbox рендерит request.title/summary — строки, которые se-pipeline уже генерит: «approve to WAIVE and continue…»). Строить надо только то, чего там нет, и это три тонких куска поверх готовых хуков: (1) push — OS-уведомление на invalidate approvals-коллекции («runs page the operator»: work идёт часами, смотреть на граф — не режим соло-оператора); (2) andon — карточка «почему запарковано» для BOUND_STALE/крашей: подсветить ребро привязки gate0 → bound-ноды с digest mismatch и легальными действиями, вместо ручного smithers why + раннбук; (3) sparkline — прогон как строка из 5 клеток-стадий (цвет = вердикт, бейдж = пауза) — плотная замена se list; полный граф — это zoom, не домашний экран.

Basis
direct: опровержение C/D верификатором (monitor.tsx:288–369 ApprovalsInbox; serializeApprovalRow маппит request.title/summary) — значит, строить надо только дельту; exit 3 = durable wait; ProofBinding/BOUND_STALE подтверждён. external: доноры из awesome-smithers: burns — approval как первоклассное состояние UI, не строка лога (+ workspace-registry на N реп, актуально при выходе на репы integration.app); svvy — lane-метафора (strategy-lane + bounded execution-lanes) для группировки sparkline по стадиям. reasoned: ритм соло-оператора — часы автономии, секунды решения; дашборд-поза чужая.
Rationale
Дельта против встроенного — то, что реально болит: узнать о паузе не открывая ничего, понять «почему стоит» за один взгляд, видеть флот в 10 строк. Каждый кусок независим и мал.
Downsides
Ценность зависит от того, чего не хватает именно тебе в monitor — сперва неделю пожить в нём (см. Rejection Summary); риск дублировать то, что smithers доделает сам (192 открытых issue, UI-темы среди них).
Confidence
65%
Complexity
Medium
